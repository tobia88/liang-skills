Pattern: Both executors share identical Campaign Intake structure: read manifest, collect planned quests, sort by dependency, show table, confirm. The workflow enforcement inserts a filtering + error-reporting layer into the queue-building step.

Three-state model: workflow matches (process), workflow mismatches (hard-block + report), workflow absent (distinct error). Hard-block is per-quest, not per-campaign.

s01 Add workflow enforcement to general executor Tier 1: Command

Add workflow filtering, mismatch hard-blocking, and planned-but-untagged detection to liang-quest-general-executor/SKILL.md’s Campaign Intake section.

Files: liang-quest-general-executor/SKILL.md

Preconditions
  • liang-quest-general-executor/SKILL.md exists
  • Campaign Intake collects all planned quests without workflow filtering
Postconditions
  • Campaign Intake filters quests by workflow: general
  • SKILL.md contains hard-block logic for workflow mismatches
  • SKILL.md contains distinct error handling for planned-but-untagged quests
  • Core Contract declares workflow enforcement responsibility
Implementation Instructions
Edit liang-quest-general-executor/SKILL.md: 1. In the Core Contract section (around lines 17-30), add a new bullet: - **Workflow enforcement:** Only process quests with workflow: "general" in the manifest. Hard-block and report quests with a mismatched workflow tag. Flag quests that are planned but have no workflow tag as a distinct error category. 2. In Step 5 (Campaign Intake), substep 2 (Build the queue, around line 113): Current: "**Build the queue** -- Collect all quests with status: planned. Sort by dependency order." Replace with: **Build the queue** -- Read all quests from the manifest. For each quest with status: planned, check the workflow field: - **workflow: "general"** -- Add to the execution queue. - **workflow is present but not "general"** (e.g., "tdd", "quick") -- **Hard-block.** Do not add to the queue. Report: "Quest <quest-id> has workflow: <value>, expected: general. Skipping -- this quest belongs to a different executor." - **workflow is absent or empty** (planned but untagged) -- **Distinct error.** Do not add to the queue. Report: "Quest <quest-id> is planned but has no workflow tag. This indicates the tactician did not stamp workflow. Run the tactician on this campaign to stamp workflow before executing." Sort the eligible quests by dependency order. Display a summary of enforcement results before the queue table: - Eligible: N quests with workflow: general - Skipped (wrong workflow): N quests (list IDs) - Error (untagged): N quests (list IDs) 3. In Step 5, substep 3 (Show the queue), add a workflow column to the queue table. 4. In the Boundaries section, add a new boundary: **Process quests with workflow other than "general".** This executor only handles general workflow quests. TDD quests belong to liang-quest-tdd-executor; quick quests belong to liang-quest-quick. 5. In the Failure Modes section, add two entries: - **Workflow mismatch detected:** Hard-block the mismatched quest. Report which executor should handle it. Continue processing eligible quests. - **Planned but untagged quest detected:** Report the error with guidance to run the tactician. Continue processing eligible quests.
Acceptance Criteria
  • Campaign Intake filters quests by workflow: general
  • Workflow mismatch produces a hard-block with report naming the correct executor
  • Planned-but-untagged quests produce a distinct error recommending the tactician
  • The two error categories are clearly distinguishable
