# Shared Asset Pool Protocol

Location: `.liang/prototypes/shared_assets/` — a sibling of every prototype folder. One
pool for the whole workspace, so every asset generated for one prototype is a candidate
for the next. The pool only pays off if the catalog stays trustworthy: read it before
generating, register immediately after generating.

## catalog.md

A single markdown table, one row per asset file. Create with just the header row if
missing.

```markdown
# Shared Asset Catalog

| file | type | size | description | tags | source |
|------|------|------|-------------|------|--------|
| player-mech-32.svg | sprite | 32x32 | Blocky bipedal mech, side view, right-facing | player, mech, side-view | 260611-1532_grapple-platformer |
| cave-tiles-16.svg | tileset | 16x16 x8 | Dark rock tiles: floor, wall, corners | tiles, cave, dark | 260611-1532_grapple-platformer |
| jump-blip.wav | sfx | 0.1s | Short rising square bleep | jump, ui, bleep | 260608-0915_dash-platformer |
```

- **type**: `sprite`, `tileset`, `background`, `sfx`, `music`, `font`.
- **size**: pixel dims for images (and frame count if a sheet), duration for audio.
- **description**: written for the *next* agent scanning for a match — say what it looks
  or sounds like, orientation, palette mood. Not how it was made.
- **tags**: lowercase, comma-separated, the primary lookup key.
- **source**: folder name of the prototype that first generated it.

## Reuse rules

1. **Read the catalog before generating any asset.** Match by tags and description.
2. **Close enough beats new.** Adapt in code — `ctx.scale`, flip with negative scale,
   tint via `ctx.filter = 'hue-rotate(...)'` or compositing — rather than generating a
   near-duplicate of an existing asset.
3. **Reference, don't copy.** Prototypes load pool files via the relative path
   `../shared_assets/<file>`. Never copy pool files into prototype folders.
4. **`file://` constraint.** Prototypes open by double-click, so load assets only
   through `Image()` / `Audio()` objects. `fetch()`, XHR, and ES module imports fail on
   `file://`. This also rules out JSON data files in the pool — bake data into the
   asset's SVG or the prototype's script.

## Generation rules

1. **SVG first for visuals.** SVGs are hand-writable, scale cleanly, and render fine via
   `drawImage`. Use PNG only when the look genuinely needs raster (noise, dithering).
   Audio: write a file only when it is distinctive enough to reuse; trivial bleeps are
   synthesized inline with WebAudio instead (see html-template.md).
2. **Descriptive kebab-case names with a size suffix**: `player-mech-32.svg`,
   `boss-crab-64.svg`, `cave-tiles-16.svg`, `laser-zap.wav`. The name should survive
   being read in a catalog two months later.
3. **Register in the same step.** Append the catalog row immediately after writing the
   file — an unregistered asset is invisible and will be regenerated by the next
   prototype.
4. **Pool assets are immutable.** Other prototypes reference them by path; editing or
   deleting one silently breaks old prototypes. To improve an asset, add
   `player-mech-32-v2.svg` and register it. Leave the old file and row alone.
5. **Generic over specific.** Where it costs nothing, lean toward reusable framing — a
   "blocky bipedal mech" serves more future prototypes than a "level-3 grapple mech with
   cooldown rings"; keep prototype-specific decoration in code overlays.

## Recovery

- `catalog.md` missing or out of sync with the directory: rebuild it — list the files,
  open each (SVGs are readable text), describe and re-register. Mark `source` as
  `unknown` where it can't be recovered.
- A referenced asset is missing at load time: the fallback-shape pattern in
  html-template.md keeps the prototype playable; note the gap in the README.
